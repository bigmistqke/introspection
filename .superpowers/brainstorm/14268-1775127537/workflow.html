How does the CLI connect to a running browser?

The core workflow question — pick the model that feels right

A

Remote Debugging Port (CDP WebSocket)

Launch Chromium with --remote-debugging-port=9222. The CLI connects to ws://localhost:9222 directly — same way VS Code attaches to Chrome. Works while the browser is alive.

Workflow
# playwright.config.ts
launch: { args: ['--remote-debugging-port=9222'] }

# terminal 1 — run tests
npx playwright test --headed

# terminal 2 — AI queries while browser is open
introspect eval "window.store.getState()" --port=9222
introspect dom --selector="#app" --port=9222
introspect network --port=9222

Pros

  • Standard Chrome protocol
  • Live arbitrary expression eval
  • No intermediary process

Cons

  • Browser must stay open for queries
  • Post-mortem requires keeping browser alive manually
  • Port conflicts in parallel test runs
B

Introspection Daemon (local proxy)

A small background process (introspect daemon) connects to the browser via CDP and streams events. Tests register with the daemon. CLI talks to the daemon — even after the browser closes, the daemon holds the trace in memory.

Workflow
# start daemon once
introspect daemon start

# tests auto-register via attach()
npx playwright test

# AI queries anytime (browser dead or alive)
introspect eval "window.store.getState()" --test=login
introspect query --type=error --test=login

Pros

  • Queries work after browser closes
  • Central place for all test sessions
  • Can serve multiple parallel tests

Cons

  • Extra process to manage
  • Live eval only while browser is up
  • More moving parts
C

Session File + On-Demand CDP Attach

When a test starts, attach(page) writes a .introspect-session file with the CDP WebSocket URL + test metadata. CLI reads the session file and connects directly. On test end, a .trace.json replaces the session file (with snapshots for post-mortem).

Workflow
# test starts → session file written
.introspect/login-test.session.json

# while test runs (live)
introspect eval "document.title" --session=login-test

# test ends → trace file written
.introspect/login-test.trace.json

# post-mortem (browser closed)
introspect query --type=error --trace=login-test

Pros

  • No daemon needed
  • Live + post-mortem in one model
  • Session file is just JSON — easy to discover
  • Scales naturally to parallel tests

Cons

  • Post-mortem eval limited to saved snapshots
  • Slightly more complexity in attach()

My recommendation: C — session file approach. Clean, no extra processes, works for both live and post-mortem. The only limitation (no arbitrary eval after browser closes) is acceptable if we take rich snapshots on errors.
