## Summary

**Root cause:** Race condition in `app.html`. The app calls `loadProfile()` immediately after initiating `fetch('/api/auth')`, without waiting for the auth response. Because `loadProfile()` is not inside the auth `.then()` chain, it fires with `token = null`, sending the profile request without an `Authorization` header — causing a 401.

**Evidence from trace:**
- `introspect summary`: 1 failed network request — `401 /api/profile`
- `introspect timeline`: `/api/profile` and `/api/auth` requests fired at the exact same timestamp (concurrent, not sequential)
- `introspect dom`: DOM snapshot showed `loadProfile()` called outside the auth promise chain

**Fix:** Move `loadProfile()` call inside the `.then()` callback that sets `token`, ensuring the profile request only fires after authentication completes.

## Post-investigation: API gap discovered

Reading the rapport revealed that `introspect timeline` showed network events with garbage timestamps: `[-1775026403717ms]`. The agent redacted these as `[-...ms]` and inferred "same timestamp / concurrent" from context — the correct conclusion, but based on broken data.

**Root cause of the broken timestamps:** `toTs()` in `packages/core/src/cdp.ts` did `timestamp * 1000 - startedAt`, where CDP network `timestamp` is a monotonic clock (seconds since Chrome started, ~1.7M seconds) and `startedAt` is Unix epoch milliseconds. These are on different clocks.

**Fix applied:**
- `normaliseCdpNetworkRequest` now uses `wallTime` (CDP's actual Unix epoch seconds) instead of `timestamp`
- `normaliseCdpNetworkResponse` takes a `cdpTimeOffset` param (computed from first request's `wallTime - timestamp`) to convert monotonic timestamps to wall-clock-relative ms
- `attach.ts` captures the offset on the first `Network.requestWillBeSent` event and passes it to `normaliseCdpNetworkResponse`

Timestamps now show correctly (e.g. `[15ms]`, `[31ms]` relative to session start).
