# Introspection Plugin System Design

**Date:** 2026-04-06
**Status:** Draft

## Overview

A plugin system for `@introspection/playwright` that enables browser-side instrumentation via CDP, without Vite, WebSocket, or a browser agent. Plugins inject browser-side scripts, push discrete events via `Runtime.addBinding`, and pull full state snapshots on demand. The system is designed for two primary consumers: developers configuring plugins upfront, and Claude adding watchpoints dynamically during live investigation sessions.

---

## Core Concepts

**Session** — what `attach()` returns. Unlimited lifetime. A Playwright test is one host; a long-running investigation script is another. The session writes to `events.ndjson` regardless of host.

**Plugin** — a self-contained object that: injects a browser-side script, exposes a typed `watch()` API, and optionally captures full state on trigger. The plugin instance is the API surface — callers hold a reference and call `watch()` directly on it.

**Subscription** — a watch rule installed by the developer or Claude. Managed browser-side via the plugin's own `watch(spec)` / `unwatch(id)` API. Each call to `watch()` returns a cleanup ID; calling `unwatch(id)` triggers the plugin's cleanup callback (ref-counted proxy removal).

**Push bridge** — `Runtime.addBinding('__introspect_push__')` creates `window.__introspect_push__` in the browser. When called from browser JS, fires `Runtime.bindingCalled` on the CDP session. Node-side handler parses the payload and calls `appendEvent`. No WebSocket, no extra server — events travel via the existing CDP connection.

`Runtime.addBinding` is a CDP session-level operation scoped to the browser context. It survives same-context page navigation and does not need to be re-issued after `Page.frameNavigated`.

## Plugin Interface

```ts
interface IntrospectionPlugin {
  name: string
  script: string                               // browser-side JS, injected via addScriptToEvaluateOnNewDocument
  install(ctx: PluginContext): Promise<void>   // called by attach(); plugin stores ctx here
  capture?(trigger: 'js.error' | 'manual' | 'detach', ts: number): Promise<CaptureResult[]>
}

interface PluginContext {
  page: Page
  cdpSession: CDPSession
  emit(event: Omit<TraceEvent, 'id' | 'ts'> & { id?: string; ts?: number }): void
  // writeAsset wraps core's writeAsset; source defaults to 'plugin' (unlike core's default of 'agent')
  writeAsset(opts: {
    kind: string
    content: string | Buffer
    ext?: string
    metadata: { timestamp: number; [key: string]: unknown }
    source?: EventSource   // default: 'plugin'
  }): Promise<string>
  timestamp(): number   // ms since session start
}

interface CaptureResult {
  kind: string                           // asset kind, e.g. 'webgl-state'
  content: string                        // JSON stringified
  summary: Record<string, unknown>       // inline summary for the asset event
}
```

`install()` is called once by `attach()` after the CDP session is open and the script has been injected. The plugin stores `ctx` internally — this is what `watch()` uses to call `Runtime.evaluate` later.

`capture()` returns `CaptureResult[]` (an array) because a single plugin may produce multiple assets per trigger — for example, one asset per WebGL context. Core calls `writeAsset` for each result. Returning an empty array is valid (no assets to write). `capture()` is optional; plugins that only push events omit it.

`ctx.emit` matches the internal convention in `attach.ts`: `id` and `ts` are optional and will be generated by core if absent.

`ctx.writeAsset` accepts an optional `source` parameter defaulting to `'plugin'` — unlike core's `writeAsset` which defaults to `'agent'`. Plugin captures are automatically filterable as `source: 'plugin'` without callers passing it explicitly.

## Browser-Side Contract

Each plugin's `script` must expose a standardised API on `window.__introspect_plugins__[name]`:

```js
window.__introspect_plugins__ = window.__introspect_plugins__ || {}
window.__introspect_plugins__.webgl = {
  watch(spec) {
    // install interceptors as needed (ref-counted)
    // store cleanup under a generated ID
    // return the ID as a string
    return id
  },
  unwatch(id) {
    // call stored cleanup for id
    // remove interceptors when ref count reaches zero
  }
}
```

`watch(spec)` returns a string ID synchronously. The core calls it via `Runtime.evaluate` and stores the returned ID. `unwatch(id)` triggers the plugin's internal cleanup.

Interceptors are **lazy** — nothing is proxied until a subscription is active. When the last subscription for a given event type is removed, the original function is restored.

## Types: `PluginEvent` and `EventSource`

`source: 'plugin'` is added to the `EventSource` union in `@introspection/types`:

```ts
type EventSource = 'cdp' | 'agent' | 'playwright' | 'plugin'
```

A new `PluginEvent` type is added to the `TraceEvent` union to accommodate plugin-pushed events with arbitrary `type` strings:

```ts
interface PluginEvent extends BaseEvent {
  source: 'plugin'
  type: string   // e.g. 'webgl.uniform', 'redux.action'
  data: Record<string, unknown>
}

// Updated TraceEvent union (packages/types/src/index.ts):
type TraceEvent =
  | NetworkRequestEvent
  | NetworkResponseEvent
  | NetworkErrorEvent
  | JsErrorEvent
  | MarkEvent
  | PlaywrightActionEvent
  | PlaywrightResultEvent
  | AssetEvent
  | PluginEvent   // ← new
```

The CLI `VALID_SOURCES` set in `packages/cli/src/commands/events.ts` is updated to include `'plugin'`.

## `attach()` Changes

```ts
interface AttachOptions {
  outDir?: string
  testTitle?: string
  workerIndex?: number
  plugins?: IntrospectionPlugin[]   // new
}
```

During `attach()`:
1. For each plugin: inject `plugin.script` via `Page.addScriptToEvaluateOnNewDocument`
2. Install `Runtime.addBinding('__introspect_push__')`
3. Wire `Runtime.bindingCalled` → parse payload → `ctx.emit` → `appendEvent`
4. Call `plugin.install(ctx)` for each plugin

---

## Push Event Format

Browser-side call: `window.__introspect_push__(JSON.stringify({ type, data }))`. The `ts` is generated Node-side using `ctx.timestamp()` at the moment `Runtime.bindingCalled` fires.

Node-side wrapping produces a full `TraceEvent`:

```ts
{
  id: <generated uuid>,
  type: payload.type,           // e.g. 'webgl.uniform'
  source: 'plugin',
  ts: ctx.timestamp(),
  data: payload.data,
}
```

The browser payload deliberately omits `ts` — browser `performance.now()` is not aligned with the session start time. Node generates `ts` on receipt.

## Subscription Lifecycle

**Adding a watch:**
1. Plugin's typed `watch()` method translates options to a serialisable `spec` object
2. Calls `Runtime.evaluate('(() => window.__introspect_plugins__[name].watch(' + JSON.stringify(spec) + '))()')` → returns ID string
3. Stores `{ plugin, spec, id }` in Node-side subscription registry
4. Returns `WatchHandle`

```ts
interface WatchHandle {
  unwatch(): Promise<void>
}
```

**Removing a watch:**
1. `watchHandle.unwatch()`
2. `Runtime.evaluate` calls `window.__introspect_plugins__[name].unwatch(id)`
3. Plugin's cleanup callback fires, ref count decremented, interceptor removed if zero
4. Entry removed from Node-side subscription registry

**Page navigation:**

Core listens on `page.on('load', ...)` (fires after the new document and its scripts have loaded, ensuring `window.__introspect_plugins__` is initialised). On each load event:
1. Old browser-side IDs are abandoned (the old context is gone; no explicit cleanup needed)
2. For each `{ plugin, spec }` in the Node-side registry: call `Runtime.evaluate` to re-call `watch(spec)`, store the new ID
3. Watches are transparently restored in the new page context

**Bulk cleanup at `detach()`:**

`detach()` calls `unwatch()` on every active `WatchHandle` in the Node-side registry before finalising the session. Plugins do not need to track handles themselves.

---

## Capture on Trigger

On `Runtime.exceptionThrown` (trigger `'js.error'`) and `handle.snapshot()` — the manual snapshot method on `IntrospectHandle` (trigger `'manual'`) — core calls `plugin.capture(trigger, ts)` for each installed plugin that implements it. For each `CaptureResult` returned, core calls `ctx.writeAsset({ ..., source: 'plugin' })` and appends the `asset` event to `events.ndjson`.

At `detach()`, core calls `plugin.capture('detach', ts)` for plugins that implement `capture`. A plugin that only wants error/manual snapshots can check `trigger !== 'detach'` and return `[]`.

---

## `source: 'plugin'` in the CLI

With `'plugin'` added to `VALID_SOURCES`, `introspect events --source plugin` returns all plugin-pushed events and plugin-generated asset events. This is the primary way Claude filters to plugin-domain data.

---

## Key Design Decisions

- **Plugin instance is the API surface** — callers hold a reference to the plugin object and call `watch()` directly. No `handle.plugins` registry, no complex TypeScript generics on `attach()`.
- **Interceptors are lazy** — nothing is proxied until subscribed; restored when last subscription is removed. Plugin manages its own ref counting.
- **Node owns subscription registry** — browser-side IDs are ephemeral (lost on navigation). Node re-applies specs after each `load` event using stored `{ plugin, spec }` pairs.
- **Push via `Runtime.addBinding`** — uses the existing CDP connection, survives navigation, no extra transport.
- **Pull via plugin-owned evaluate calls** — `capture()` is a Node-side method on the plugin object; it may use `ctx.page.evaluate()` or `ctx.cdpSession.send('Runtime.evaluate', ...)` internally — the plugin's choice. Full state is written as sidecar assets with `source: 'plugin'`.
- **`ts` generated Node-side** — browser `performance.now()` is not aligned with session start; timestamp is assigned when `Runtime.bindingCalled` fires.
- **`detach()` bulk-unwatches** — callers do not need to clean up individual `WatchHandle`s before detaching.
- **`capture()` returns array** — supports plugins that produce multiple assets per trigger (e.g. one per WebGL context).
