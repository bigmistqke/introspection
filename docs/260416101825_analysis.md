## Not Fixed â€” Recommended Follow-ups

### TOCTOU in `initTraceDir` (`packages/write/src/trace-writer.ts:15`)

```ts
const exists = await stat(traceDir).then(() => true, () => false)
if (exists) throw new Error(`Trace directory already exists: ${traceDir}`)
await mkdir(join(traceDir, 'assets'), { recursive: true })
```

The `stat` check and `mkdir` are not atomic. Another process could create the directory between the two calls, making the guard unreliable. The idiomatic fix: skip the `stat`, call `mkdir` without `recursive`, and catch `EEXIST` to distinguish "already existed" from other errors. Behaviour change risk is low but the test suite would need updating.

### TOCTOU in `packages/serve/src/handler.ts`

Multiple `existsSync()` guards precede `readdirSync`/`readFileSync`/`createReadStream` calls. If the filesystem changes between the check and the operation, the subsequent call throws anyway â€” making the guards misleading rather than protective. Replace with direct operations + error handling.

```ts
// Before
if (!existsSync(eventsPath)) return new Response('', { ... })
const stat = statSync(eventsPath)
const stream = createReadStream(eventsPath)

// After: just try, catch ENOENT
```

### SSE handler low-level fd management (`packages/serve/src/handler.ts:108â€“145`)

The SSE change-detection loop manually opens/reads/closes a file descriptor on every `fsWatch` event:

```ts
const fd = openSync(eventsPath, 'r')
const buffer = Buffer.alloc(stat.size - position)
readSync(fd, buffer, 0, buffer.length, position)
closeSync(fd)
```

This is synchronous I/O on the event loop and the fd could leak if an exception is thrown mid-read. `fs.createReadStream({ start: position })` is the idiomatic replacement and handles buffering correctly.

### Adapter branching leaked into all operations (`packages/write/src/trace.ts`)

Every operation in `createTraceWriter` (`emit`, `writeAsset`, `finalize`) contains an `if (adapter) { ... } else { ... }` branch. The adapter abstraction is resolved at each call site rather than once at construction time. The clean fix is to resolve a unified internal write interface at the top of `createTraceWriter` that exposes `appendLine`, `writeFile`, and `finalizeTrace` â€” then the rest of the body is branch-free. This is a larger refactor (~50 lines) with a good test surface.

### Bus has no `off()` mechanism (`packages/utils/src/bus.ts`)

```ts
on<T>(trigger: T, handler: BusHandler<T>) {
  const existing = handlers.get(trigger) ?? []
  existing.push(handler)
  handlers.set(trigger, existing)
}
```

Handlers accumulate unboundedly. In long-running processes (persistent test servers, CI runners creating many traces), this is a latent memory and CPU leak â€” every emit walks a growing handler list. Adding `off()` or returning an unsubscribe function from `on()` would fix it. Requires interface change on `TraceBus` in types.

### `assets.ls()` O(n) full scan (`packages/read/src/index.ts:177â€“182`)

Iterates all events on every call. Not a problem at current scale, but if called in a tight polling loop or UI that refreshes frequently, it re-scans the entire event stream. Cache-on-push would eliminate the cost.
