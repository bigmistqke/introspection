# Codebase Analysis â€” Simplification & Workflow Improvements

_Date: 2026-04-03_

## Summary

The introspection monorepo is architecturally clean and well-typed at its core, with a clear separation of concerns across the Vite plugin hub, Playwright attach layer, CLI query layer, and plugin system. The main themes of friction are: (1) several `CaptureConfig` fields are documented but unimplemented, creating false expectations; (2) the browser-side plugin activation path is entirely undocumented and requires a separate package (`@introspection/browser`) nobody is told about; (3) a handful of event types defined in `@introspection/types` are never emitted by any package, creating dead schema surface. There are also concrete bugs in the network command's requestâ€“response join logic and the trace-writer silently discarding Playwright's reported test duration.

## Findings

### [Dead Config]: CaptureConfig network and body fields are documented but never implemented

**Severity:** High
**Files:**
- `packages/types/src/index.ts` lines 201â€“211 (`CaptureConfig`)
- `packages/vite/src/server.ts` line 68
- `README.md` lines 143â€“149

**Problem:** `CaptureConfig` declares `network.ignoreUrls`, `network.ignoreHeaders`, `responseBody.maxSize`, and `responseBody.ignore`. The README documents all of them with usage examples. In `server.ts`, only `config.capture?.ignore` (event type suppression) is actually read â€” the other four fields are silently ignored. No response body is ever fetched or stored anywhere in the codebase (the `Session.bodyMap` field exists but is never populated). Users who configure these will get no error and no effect.

**Suggestion:** Either implement the fields (add `Network.getResponseBody` CDP call in `attach.ts`, add URL/header filtering in `server.ts`), or remove them from the type and README until they are ready. At minimum add a `// TODO: not yet implemented` comment in the type definition.

### [Bug]: `network` command requestâ€“response join always fails

**Severity:** High
**Files:**
- `packages/cli/src/commands/network.ts` lines 7â€“9, 19
- `packages/playwright/src/cdp.ts` lines 4, 13, 29

**Problem:** `formatNetworkTable` builds a `Map<e.id, request>` where `e.id` is the random event ID generated by `makeId()` (e.g. `evt-3a7c1f2b`). It then looks up `requests.get(res.data.requestId)` where `res.data.requestId` is the CDP-assigned numeric request ID (e.g. `"1234.5"`). These two values are never equal, so `req` is always `undefined` and the METHOD column always shows `?`. The unit test masks this bug by using the same value for both `e.id` and `data.requestId`, which doesn't reflect real trace data.

**Suggestion:** In `normaliseCdpNetworkRequest`, add `data.cdpRequestId: raw.requestId as string` and key the map on that. Or key the requests map on `e.data.url` as a fallback. Update the test to use realistic data.

### [Missing Affordance]: Browser-side plugin activation path is completely undocumented

**Severity:** High
**Files:**
- `README.md` lines 87â€“113
- `packages/browser/src/index.ts`
- `packages/cli/skills/introspect-setup/skill.md`
- `packages/cli/skills/introspect-plugin/skill.md`

**Problem:** The `browser.setup(agent)` method on `IntrospectionPlugin` is only ever called from `BrowserAgent.use()` in `@introspection/browser`. The README's plugin sections show only the Vite-config (server-side) registration. There is no mention anywhere of `BrowserAgent.connect()`, `agent.use()`, or when to import `@introspection/browser`. A user following the README for `createReduxPlugin` or `createReactPlugin` will correctly register server-side hooks but the browser-side `setup()` (which installs the actual monkey-patches on `store.dispatch`, `__REACT_DEVTOOLS_GLOBAL_HOOK__`, etc.) will never run, producing no events.

**Suggestion:** Add a "Browser-side wiring" section to the README and `introspect-setup` skill explaining that `@introspection/browser` must be imported in the app's entry point, with `BrowserAgent.connect(url, sessionId)` and `agent.use(plugin)` called. Also update `introspect-plugin` skill.

### [Bug]: `trace-writer` silently discards Playwright's reported test duration

**Severity:** Medium
**Files:**
- `packages/vite/src/trace-writer.ts` line 71
- `packages/types/src/index.ts` line 147 (`TestResult`)

**Problem:** `writeTrace` receives a `TestResult` with `result.duration` (from `testInfo.duration` in the fixture â€” the authoritative Playwright-measured value). Instead of using it, line 71 computes `Date.now() - session.startedAt`. Since `session.startedAt` is set when `startSession` is received by the Vite server (after a round-trip over WebSocket), and the session ends when `endSession` is received, this measures server-side time including network latency, not actual test time. The passed `result.duration` is read for `status` and `error` but never for `duration`.

**Suggestion:** Replace `duration: Date.now() - session.startedAt` with `duration: result.duration` and remove `startedAt` from `Session` if it's no longer needed for other purposes.

### [Dead Code]: Five event types defined in types but never emitted

**Severity:** Medium
**Files:**
- `packages/types/src/index.ts` lines 39â€“71

**Problem:** `DomSnapshotEvent` (`dom.snapshot`), `VariableSnapshotEvent` (`variable.snapshot`), `BrowserClickEvent` (`browser.click`), `BrowserInputEvent` (`browser.input`), and `JsConsoleEvent` (`js.console`) are defined as concrete interfaces in `TraceEvent` but are never emitted by any package. The debug skill documents `dom.snapshot` and `variable.snapshot` in its event type table as if they are live, which is misleading. `browser.click` and `browser.input` suggest more granular tracking than the current `playwright.action` proxy provides.

**Suggestion:** Either implement the missing emitters (e.g. emit `js.console` in `attach.ts` via `Runtime.consoleAPICalled` CDP event), or remove these types and update the skill/docs. At minimum mark them with `// not yet emitted` comments.

### [Missing Affordance]: `introspect network --failed` silently omits `network.error` events

**Severity:** Medium
**Files:**
- `packages/cli/src/commands/network.ts` lines 5â€“23
- `packages/cli/src/trace-reader.ts` lines 33â€“34

**Problem:** `formatNetworkTable` only processes `network.response` events. `network.error` events (DNS failure, connection refused, timeout) are structurally different â€” they have no status code and no matching request entry. When `--failed` is passed, `network.error` events are ignored entirely. An AI or user running `introspect network --failed` after a connectivity failure will see an empty table even though `network.error` events exist.

**Suggestion:** Extend `formatNetworkTable` to also include `network.error` events, rendered with a synthetic status like `ERR` and the `errorText` field as the URL column.

### [CLI UX]: `--at` flag on `vars` command is accepted but not implemented

**Severity:** Medium
**Files:**
- `packages/cli/src/index.ts` line 42
- `packages/cli/src/commands/vars.ts`

**Problem:** `program.command('vars').option('--at <point>')` registers the flag in Commander but `formatVars` never receives or uses `opts.at`. A user attempting `introspect vars --at 500` gets no error and no change in output, with no indication the flag does nothing. A prior commit removed it from the debug skill but it's still wired in the CLI.

**Suggestion:** Either implement time-point selection (find the `variable.snapshot` event nearest to `--at <ms>`) or remove the `--at` option from the command declaration to prevent silent no-ops.

### [CLI UX]: No `--dir` flag; CLI is hardcoded to `.introspect`

**Severity:** Medium
**Files:**
- `packages/cli/src/index.ts` lines 16, 23, 58, 65

**Problem:** `DEFAULT_OUT_DIR` is resolved once at startup from `process.cwd()/.introspect`. Every command uses it without an override flag. If a project uses a custom `outDir` in `vite.config.ts` (e.g. `outDir: 'test-results/traces'`), every `introspect` command fails with "No trace files found" and there is no flag to point it elsewhere.

**Suggestion:** Add a `--dir <path>` global option to the root `program` command, falling back to `DEFAULT_OUT_DIR`. Wire it into `loadTrace` and `evalExpression`.

### [Package Coupling]: `@introspection/playwright` imports `@introspection/vite/snapshot` across package boundary

**Severity:** Medium
**Files:**
- `packages/playwright/src/attach.ts` lines 12â€“13
- `packages/playwright/package.json`
- `packages/vite/package.json` lines 10â€“13

**Problem:** `attach.ts` imports `takeSnapshot` from `@introspection/vite/snapshot` with a `// @ts-expect-error Cannot resolve path to vite snapshot`. This works at runtime but fails TypeScript resolution in the unbuilt monorepo because the vite package's `./snapshot` export points to `./dist/snapshot.d.ts`. It is also architecturally odd â€” the Playwright layer (which runs in Node/CI) importing a Vite-plugin internal.

**Suggestion:** Move `takeSnapshot` to `@introspection/types` or a separate `@introspection/cdp` utility package. Remove the `@ts-expect-error`.

### [Type Gap]: CDP normalizers use `Record<string, unknown>` casts throughout

**Severity:** Low
**Files:**
- `packages/playwright/src/cdp.ts` lines 10â€“62

**Problem:** `normaliseCdpNetworkRequest`, `normaliseCdpNetworkResponse`, and `normaliseCdpJsError` all accept `raw: Record<string, unknown>` (typed as `never` at call sites) and cast every field access with `as string`, `as number`, `as Record<string, string>` etc. Type errors in the normalizers (wrong field name, wrong nesting) won't be caught by the compiler.

**Suggestion:** Define typed interfaces for the CDP raw event shapes (or import from `@playwright/test` if available) and remove the `Record<string, unknown>` cast at the function boundary.

### [Missing Affordance]: `browser.navigate` event always has an empty `from` field

### [Missing Affordance]: `eval` socket picks sessions by array index, breaking under parallel workers

**Severity:** Low
**Files:**
- `packages/vite/src/eval-socket.ts` lines 32â€“33

**Problem:** `const session = all[all.length - 1]` assumes the last session in insertion order is the relevant one. With parallel Playwright workers, the "last" session depends on connection order, not test identity. There is no way to target a specific session by ID or title.

**Suggestion:** Accept an optional `sessionId` field in the eval message protocol, defaulting to the most recently started session.

### [Package Coupling]: `@introspection/playwright` lists `@introspection/vite` as a runtime dependency

**Severity:** Low
**Files:**
- `packages/playwright/package.json`

**Problem:** Any project that installs `@introspection/playwright` also transitively installs the full Vite plugin and its dependencies (`ws`, `source-map`, `@bigmistqke/rpc`). A Playwright-only consumer (e.g. in a CI environment without a Vite dev server) pulls in unnecessary code.

**Suggestion:** Resolved once `takeSnapshot` is moved to a shared package (see cross-package import finding). In the interim, document it as a known transitive cost.

### [Inconsistency]: Package `types` exports are inconsistent across the monorepo

**Severity:** Low
**Files:**
- `packages/playwright/package.json`: `"types": "./src/attach.ts"`
- `packages/vite/package.json`: `"types": "./dist/index.d.ts"`
- All other packages: `"types": "./dist/index.d.ts"`

**Problem:** The playwright package uses raw TypeScript source for type resolution (following commit `a749729`), while all other packages point to `./dist/*.d.ts`. This means the monorepo works without building playwright but breaks without building vite, browser, and types packages. The inconsistency is the root cause of the `@ts-expect-error` on the `vite/snapshot` import.

**Suggestion:** Standardize to `"types": "./src/index.ts"` across all packages for unbuilt monorepo use, or document that `pnpm build` is required before TypeScript resolution works.
